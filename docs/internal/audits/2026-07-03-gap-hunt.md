PR #23's room core is largely sound — the room-resolution helpers
(`channelRoomId` / `nodeRoomIds` / `resolveSelectionRoom`), the per-level scope-target
expansion (`addNodeScopeTargets` union vs `addNodeResourceTargets` single-room), the
meter per-channel filter (`filterMeterFrameForUser`), and the ad-hoc recording and
schedule **create** paths all correctly enforce the shared-node isolation invariant.
The confirmed gaps clustered where a request or a later event **changes the room a
resource resolves to** — schedule update, channel reassignment, scheduled-run time —
and one authorization skip on the room-less edge.

| R1-SSE | Low | suspected / by-design | `node-routes.ts:752` (`GET /meter-events`) | The SSE meter stream applies no per-channel `filterMeterFrame`, unlike the single-frame `/meters` route. Verified **not** a reachable leak: the SSE gate is `hasResourceScope(node)` (roster-blind), so only owner/admin/direct-node-grant principals ever receive frames, and for them `meterRoomAccess` is `"all"` — the filter would be a provable no-op. | R1-SHRINK | Low | coverage | `node-inventory-reconcile.ts:193` / `:277` | On a channelCount **decrease**, both reconcile paths drop the per-channel room assignment of the vanished channels (intentional — the agent owns hardware truth), and a later re-grow re-creates them unassigned. Behavior is correct but untested in either store; a transient device-enumeration hiccup silently loses an operator's assignment. Add a shrink+regrow test to lock the contract. |

## Verified clean (checked, no bug)

Confirmed correct by direct source review and adversarial verification during Run 1:
the per-channel room-resolution helpers `channelRoomId` / `nodeRoomIds` /
`interfaceRoomIds` / `resolveSelectionRoom` (fallback to node default, cross-room
`ok:false`, empty-selection handling); the per-level scope-target expansion in
`scope-targets.ts` / `resource-scope-targets.ts` (a channel resolves to its own room,
an interface/node to the union, and single-room resources use `addNodeResourceTargets`
so a node grant authorizes without widening the room scope — no `"room"` branch means a
bare room target does not over-expand); the room-capability roster mapping
(`roomCapabilityAuthorizes` only grants when the target resolves to a rostered room);
the meter per-channel filter `filterMeterFrameForUser` (drops other rooms' channel
levels); the ad-hoc recording **create** path (already re-authorizes against the
resolved capture room and rejects cross-room selections); the schedule **create** path
(gates `requirePermission` on the channel-resolved room, rejects cross-room); and DB
migration `0040` / `schema.ts` for the room columns (no schema↔migration drift found).
