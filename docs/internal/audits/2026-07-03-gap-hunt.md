repeatable loop in [`docs/contributing/audit-workflow.md`](../../contributing/audit-workflow.md):

**Status legend:** `FIXED` = failing test + fix landed on this branch · `RESOLVED`
= closed indirectly by another fix · `CATALOGUED` = confirmed/reproduced, fix
recommended but not applied (blast radius, product call, or hardware verification
needed) · `SUSPECTED` = strong lead, not fully substantiated · `COVERAGE` = no bug,
a missing-test gap.

PR #23's room core is largely sound — the room-resolution helpers
(`channelRoomId` / `nodeRoomIds` / `resolveSelectionRoom`), the per-level scope-target
expansion (`addNodeScopeTargets` union vs `addNodeResourceTargets` single-room), the
meter per-channel filter (`filterMeterFrameForUser`), and the ad-hoc recording and
schedule **create** paths all correctly enforce the shared-node isolation invariant.
The confirmed gaps clustered where a request or a later event **changes the room a
resource resolves to** — schedule update, channel reassignment, scheduled-run time —
and one authorization skip on the room-less edge.

## Open work

### Catalogued / suspected

| ID | Sev | Kind | Locus | Note |
| --- | --- | --- | --- | --- |
| R3-FK-RACE | Low | catalogued | `room-store.ts` (`PostgresRoomStore.delete`) | The room-delete route's 409-in-catch branch is unreachable on Postgres because `failover` wraps FK (SQLSTATE 23503) errors as `DatabaseUnavailableError`, so a check→delete FK race 503s (self-heals on retry via the pre-check) instead of 409. Comment corrected in Run 3; the precise 409 needs `PostgresRoomStore.delete` to detect SQLSTATE 23503 and throw a distinct constraint error — not verifiable in the no-DB harness. |
| R3-OIDC-GROUP | High | suspected | `oidc-sync.ts:72` (`groupsFromIds`) | The OIDC `groups` claim ids are fed into the same id space as first-party access-group slugs, and group membership feeds `effectiveCapabilities`/`roomsForSubject`, so an IdP group id colliding with a first-party group slug could inherit that group's room-roster capabilities. Depends on the (org-controlled) IdP trust model; fails safe for deny. Namespacing OIDC group ids is a product/trust-model decision (akin to prior G18). |
| R3-GROUP-ROSTER-COV | Med | coverage | `auth-group-routes.ts:273` / `room-roster-store.ts:327` | Access-group delete **does** cascade-clean the room roster (`removeGroupFromRoster` → `removeGroupSubject`, verified wired + called), but there is no test asserting the group-delete → roster cascade, so a regression would be silent (the room-side twin of the tested Fix `G2`). Add a group-delete roster-cleanup test. |
| R5-OVERVIEW-COV | Med | coverage | `room-routes.ts:344` (`GET /rooms/:id/overview`) | The overview route is **correctly** scoped (verified): `node:read` gated on the room target, recent recordings filtered by `recording.roomId === roomId`, upcoming occurrences by `schedule.roomId === roomId` — no cross-room leak. But no test hits `/overview`, and it uses a bespoke inline `recording.roomId === roomId` filter rather than `scopedRecordings`, so a future refactor (e.g. node-membership filter, or a looser gate target) could silently leak room-B recordings/occurrences to a room-A operator on a shared node. Add a shared-node two-room overview isolation test. |
| R3-ROSTER-PUT-COV | Low | coverage | `room-routes.ts:440` (roster PUT) | `PUT /rooms/:roomId/roster` is `auth:manage`-gated + validates the room exists, then `replaceManual`. It does not validate that each subject id exists (a bogus subject just yields an inert roster row matching no one). No test covers subject-existence or the room-scope of the PUT. Low; fail-inert. |
| R2-PRECEDENCE | High | coverage | `index.ts:254` (`permissionDecision` deny-precedence) | The single guard `scope.reason !== "access_policy_denied"` is the entire mechanism ensuring an access-policy DENY beats a roster/calendar capability grant. The code is **correct**, but no test drives the app as a roster-only principal to lock it — a regression flipping the guard would silently let a roster grant override an explicit DENY with every test green. Deferred (heavy full-app harness: create user + room + roster grant + re-login + deny policy). Recipe: extend `auth-access-routes.test.ts` (already drives `app` + `loginToken`) — seed a roster-only operator, assert a room-channel action is 200 via roster (`grantedViaRoomCapability=true`), then add a user- and a group-scoped DENY and assert 403 `access_policy_denied`. |
| R2-IFACE-ENV | Low | catalogued | `schedule-route-helpers.ts:171` (`resolveScheduleRoom`) | `resolveScheduleRoom` picks `captureInterfaceId ?? node.interfaces[0]?.id` with **no** `RAKKR_AGENT_CAPTURE_INTERFACE_ID` env fallback, while the runtime capture path (`queueScheduledRecordings`, `recording-job-targets`) uses the env fallback. If a schedule pins no interface and that env var points at a non-first interface owned by a different room, the persisted `roomId`/scope diverges from the room actually captured. Narrow (requires the global env var set to a non-first cross-room interface). Fix: centralize an "effective capture interface" helper used by both, and correct the stale comment at `:165`. |
| R2-RECONCILE-ATOMICITY | Low | catalogued | `channel-room-routes.ts:128` (`reconcileNodeSchedules`) | The node channel-rooms are committed before the schedule reconcile loop, which is not wrapped in try/catch; a mid-loop `scheduleStore.update` throw leaves the node updated but some schedules stale. Self-healing (idempotent on a retry PUT; reconcile re-runs over all node schedules) and no transaction boundary exists in the handler by design. Fix: best-effort per-schedule catch inside the loop + a reconcile-failed audit note. |
| R1-SSE | Low | suspected / by-design | `node-routes.ts:752` (`GET /meter-events`) | The SSE meter stream applies no per-channel `filterMeterFrame`, unlike the single-frame `/meters` route. Verified **not** a reachable leak: the SSE gate is `hasResourceScope(node)` (roster-blind), so only owner/admin/direct-node-grant principals ever receive frames, and for them `meterRoomAccess` is `"all"` — the filter would be a provable no-op. Also `RAKKR_DEMO_METERS`-gated (no real frames in production). Cosmetic consistency nit; left unfixed to avoid dead code. |
| R1-SHRINK | Low | coverage | `node-inventory-reconcile.ts:193` / `:277` | On a channelCount **decrease**, both reconcile paths drop the per-channel room assignment of the vanished channels (intentional — the agent owns hardware truth), and a later re-grow re-creates them unassigned. Behavior is correct but untested in either store; a transient device-enumeration hiccup silently loses an operator's assignment. Add a shrink+regrow test to lock the contract. |

## Verified clean (checked, no bug)

Confirmed correct by direct source review and adversarial verification during Run 1:
the per-channel room-resolution helpers `channelRoomId` / `nodeRoomIds` /
`interfaceRoomIds` / `resolveSelectionRoom` (fallback to node default, cross-room
`ok:false`, empty-selection handling); the per-level scope-target expansion in
`scope-targets.ts` / `resource-scope-targets.ts` (a channel resolves to its own room,
an interface/node to the union, and single-room resources use `addNodeResourceTargets`
so a node grant authorizes without widening the room scope — no `"room"` branch means a
bare room target does not over-expand); the room-capability roster mapping
(`roomCapabilityAuthorizes` only grants when the target resolves to a rostered room);
the meter per-channel filter `filterMeterFrameForUser` (drops other rooms' channel
levels); the ad-hoc recording **create** path (already re-authorizes against the
resolved capture room and rejects cross-room selections); the schedule **create** path
(gates `requirePermission` on the channel-resolved room, rejects cross-room); and DB
migration `0040` / `schema.ts` for the room columns (no schema↔migration drift found).

### Notes on the runs

- **Run 1** targeted PR #23's fresh surface with six read-only hunters (room RBAC/scope,
  recording attribution, schedule attribution, node channel-room persistence,
  shared-contracts/DB, and regression blast-radius) plus an adversarial verify pass on
  every lead. 13 leads → 11 survived verification → deduped to 5 distinct confirmed bugs
  (the schedule-PATCH escalation `A` was independently found by three hunters). All 5
  fixed with red→green proof. New **SUSPECTED/coverage** leads surfaced and logged, not
  fixed: `R1-SSE` (meter-events SSE filter — cosmetic, not a reachable leak), `R1-SHRINK`
  (reconcile-shrink room-assignment drop — untested intentional behavior), and `R1-PG`
  (Postgres channel-room round-trip coverage). Dirty (landed 5 fixes) → streak 0.
- **Run 2** turned the adversary on Run 1's own fixes plus roster/access-policy, room
  lifecycle/cascade, and web room boundaries (four hunters + verify). 12 leads → 11
  survived → deduped to 6 confirmed fixes. The headline is `F`: Run 1's Fix `A` left a
  room-less hole (it only re-checked when the new room was *set*), so the adversary pass
  paid off directly. Room-delete parity (`G1`/`G2`/`G3`) and the web roster picker (`H`)
  were fresh. Also hardened my own Run-1 additions (`R2-DEDUPE` health-event flood,
  `R2-INVAL` client invalidation). New items logged, not fixed: `R2-PRECEDENCE` (High
  coverage — roster→permission deny-precedence is correct but untested; heavy harness
  deferred with a recipe), `R2-IFACE-ENV` and `R2-RECONCILE-ATOMICITY` (Low, catalogued).
  Dirty (landed 6+ fixes) → streak 0.
- **Run 3** turned the adversary on Run 2's fixes plus roster PUT / access-groups / OIDC,
  the occurrence + recording-room lifecycle, and residual web pickers (four hunters).
  The hunt found 8 leads, but a **session limit killed 6 verify agents mid-run**; the 2
  fully-verified survivors were `J` (room-detail hooks crash — HIGH) and `K` (the G3
  FK-race residual — Low). The 6 dropped leads were then verified **directly against
  source** (main-loop): `L` (move-occurrence roster reconcile — Low, fixed) held up; the
  group-delete→roster cascade was found already wired + called (`auth-group-routes.ts:273`),
  so it plus the OIDC-group-collision and roster-PUT leads are catalogued
  (`R3-*`, coverage/suspected), not bugs. `J` also motivated enabling
  `react-hooks/rules-of-hooks` repo-wide (a permanent guard for the whole class).
  Dirty (landed J, K, L) → streak 0.
- **Run 4** was a lean **hunt-only** sweep (5 hunters, no verify fan-out — the main loop
  verified, after Run 3's verify phase hit a session limit) over the **consumption** side
  (recording read/download/delete/export/metadata, health, metrics), a systematic RBAC
  route re-sweep, agent-facing routes, and a completeness critic. All five hunters
  converged on a SINGLE confirmed bug — `M`, the listen-monitor whole-node audio leak —
  and found nothing else across the consumption/RBAC/agent/completeness surface (a strong
  hardening signal). `M` had been flagged by an earlier audit but never landed; this run
  closed it (per-room monitor gate + `meter-room-access.ts` extraction). Dirty (landed M)

  node-scope, including a room-derived one, as whole-node authority). This predated `M`
