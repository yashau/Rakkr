**Status legend:** `FIXED` = failing test + fix landed on this branch · `RESOLVED`
= closed indirectly by another fix · `CATALOGUED` = confirmed/reproduced, fix
recommended but not applied (blast radius, product call, or hardware verification
needed) · `SUSPECTED` = strong lead, not fully substantiated · `COVERAGE` = no bug,
a missing-test gap.

PR #23's room core is largely sound — the room-resolution helpers
(`channelRoomId` / `nodeRoomIds` / `resolveSelectionRoom`), the per-level scope-target
expansion (`addNodeScopeTargets` union vs `addNodeResourceTargets` single-room), the
meter per-channel filter (`filterMeterFrameForUser`), and the ad-hoc recording and
schedule **create** paths all correctly enforce the shared-node isolation invariant.
The confirmed gaps clustered where a request or a later event **changes the room a
resource resolves to** — schedule update, channel reassignment, scheduled-run time —
and one authorization skip on the room-less edge.

## Open work

### Catalogued / suspected

| ID | Sev | Kind | Locus | Note |
| --- | --- | --- | --- | --- |
| R2-PRECEDENCE | High | coverage | `index.ts:254` (`permissionDecision` deny-precedence) | The single guard `scope.reason !== "access_policy_denied"` is the entire mechanism ensuring an access-policy DENY beats a roster/calendar capability grant. The code is **correct**, but no test drives the app as a roster-only principal to lock it — a regression flipping the guard would silently let a roster grant override an explicit DENY with every test green. Deferred (heavy full-app harness: create user + room + roster grant + re-login + deny policy). Recipe: extend `auth-access-routes.test.ts` (already drives `app` + `loginToken`) — seed a roster-only operator, assert a room-channel action is 200 via roster (`grantedViaRoomCapability=true`), then add a user- and a group-scoped DENY and assert 403 `access_policy_denied`. |
| R2-IFACE-ENV | Low | catalogued | `schedule-route-helpers.ts:171` (`resolveScheduleRoom`) | `resolveScheduleRoom` picks `captureInterfaceId ?? node.interfaces[0]?.id` with **no** `RAKKR_AGENT_CAPTURE_INTERFACE_ID` env fallback, while the runtime capture path (`queueScheduledRecordings`, `recording-job-targets`) uses the env fallback. If a schedule pins no interface and that env var points at a non-first interface owned by a different room, the persisted `roomId`/scope diverges from the room actually captured. Narrow (requires the global env var set to a non-first cross-room interface). Fix: centralize an "effective capture interface" helper used by both, and correct the stale comment at `:165`. |
| R2-RECONCILE-ATOMICITY | Low | catalogued | `channel-room-routes.ts:128` (`reconcileNodeSchedules`) | The node channel-rooms are committed before the schedule reconcile loop, which is not wrapped in try/catch; a mid-loop `scheduleStore.update` throw leaves the node updated but some schedules stale. Self-healing (idempotent on a retry PUT; reconcile re-runs over all node schedules) and no transaction boundary exists in the handler by design. Fix: best-effort per-schedule catch inside the loop + a reconcile-failed audit note. |
| R1-SSE | Low | suspected / by-design | `node-routes.ts:752` (`GET /meter-events`) | The SSE meter stream applies no per-channel `filterMeterFrame`, unlike the single-frame `/meters` route. Verified **not** a reachable leak: the SSE gate is `hasResourceScope(node)` (roster-blind), so only owner/admin/direct-node-grant principals ever receive frames, and for them `meterRoomAccess` is `"all"` — the filter would be a provable no-op. Also `RAKKR_DEMO_METERS`-gated (no real frames in production). Cosmetic consistency nit; left unfixed to avoid dead code. |
| R1-SHRINK | Low | coverage | `node-inventory-reconcile.ts:193` / `:277` | On a channelCount **decrease**, both reconcile paths drop the per-channel room assignment of the vanished channels (intentional — the agent owns hardware truth), and a later re-grow re-creates them unassigned. Behavior is correct but untested in either store; a transient device-enumeration hiccup silently loses an operator's assignment. Add a shrink+regrow test to lock the contract. |

## Verified clean (checked, no bug)

Confirmed correct by direct source review and adversarial verification during Run 1:
the per-channel room-resolution helpers `channelRoomId` / `nodeRoomIds` /
`interfaceRoomIds` / `resolveSelectionRoom` (fallback to node default, cross-room
`ok:false`, empty-selection handling); the per-level scope-target expansion in
`scope-targets.ts` / `resource-scope-targets.ts` (a channel resolves to its own room,
an interface/node to the union, and single-room resources use `addNodeResourceTargets`
so a node grant authorizes without widening the room scope — no `"room"` branch means a
bare room target does not over-expand); the room-capability roster mapping
(`roomCapabilityAuthorizes` only grants when the target resolves to a rostered room);
the meter per-channel filter `filterMeterFrameForUser` (drops other rooms' channel
levels); the ad-hoc recording **create** path (already re-authorizes against the
resolved capture room and rejects cross-room selections); the schedule **create** path
(gates `requirePermission` on the channel-resolved room, rejects cross-room); and DB
migration `0040` / `schema.ts` for the room columns (no schema↔migration drift found).
