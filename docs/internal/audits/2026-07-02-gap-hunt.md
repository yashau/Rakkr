# Rakkr ‚Äî Gap Hunt Findings

Adversarial correctness/security audit of the Rakkr monorepo, run as a repeatable
loop of independent read-only hunters plus adversary passes on prior fixes. Every
confirmed finding was reproduced with a failing test, then closed with a minimal
fix and re-verified green.

**Status legend:** `FIXED` = failing test + fix landed on this branch ¬∑ `RESOLVED`
= closed indirectly by another fix ¬∑ `CATALOGUED` = confirmed/reproduced, fix
recommended but not applied (blast radius, product call, or Linux-hardware
verification needed) ¬∑ `SUSPECTED` = strong lead, not fully substantiated ¬∑
`COVERAGE` = no bug, a missing-test gap.

The guardrails are genuinely strong: the authz core (deny-precedence, cascade,
token-auth atomicity, AES-256-GCM construction, key non-exposure) and the
scheduler's UTC/DST core were checked and found **correct**. Confirmed gaps
clustered where behaviour can only be proven against real hardware, Postgres, or
wall-clock time ‚Äî exactly where the structural string-presence verifiers are
blind. The audit converged on the from-Windows-testable surface; its only remaining open
*bugs* were a Linux-hardware capture-recovery slice ‚Äî now **closed in Run 28** once
the X32 rig became reachable (see [Top open item](#top-open-item)).

## Open work

<a id="top-open-item"></a>
### Top open item ‚Äî capture-recovery segment lifecycle (RS1 + GH-1 + G62 CLOSED in Run 28; GH-2 Low product call remains)

The recorder agent's recovered-segment lifecycle used to be wired **only for the
stitch-success path**. Once the X32 rig became reachable, **Run 28** closed the
three data-loss/hang gaps with red‚Üígreen proof (rig `cargo test`, real-X32 ffmpeg
concat, and fake-controller RS1/GH-1 scenarios) ‚Äî commit `da2ecd4f`:

- **RS1 (High) ‚Äî FIXED.** Restart *before* the stitch. Restart recovery now
  reconstructs the preserved segments (`runtime_segment`) and, when their files are
  still on disk, runs `stitch_recovered_capture_segments` before upload, uploading
  the stitched recording instead of only the final segment. It skips a
  `rendered`/`upload_pending` restart (segments already consumed) and persists the
  stitched path so a second restart retains the right file; on stitch failure it
  preserves the inputs + fails (never silent-completes).
- **GH-1 (High) ‚Äî FIXED.** Stitch failure during *graceful* completion.
  `stitch_recovered_capture_segments` now returns `StitchOutcome`; on concat failure
  it returns `Failed { preserved }` (every input kept on disk) and `controller.rs`
  emits a critical `capture_segments_unrecoverable` event and fails the job ‚Äî no
  more silent `completed` on the last segment alone.
- **G62 / Rust-C2 (Med) ‚Äî FIXED.** Final-chunk render failure (or no trailing
  chunk) never delivered `chunkTotal`. `finalize_and_upload_chunk` now reports
  delivery and `chunked_finish_action` finalizes the recording (`partial`/`failed`)
  when the total is undelivered and nothing is pending, instead of leaving it hung
  until lease expiry.
- **GH-2 (Low, product call) ‚Äî still open.** Restart-recovery uploads
  (`recording_job_recovery.rs`) send the raw as `rendition=None` (legacy primary),
  so a recovered recording ends up with only the raw as its single playable
  rendition (no enhanced toggle). Raw is fully preserved; consistency degradation.
  Warrants a product decision alongside G9.
